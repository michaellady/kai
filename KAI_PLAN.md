# Kai — Complete Project Plan

## Vision

A personal knowledge compounding service that ingests years of driving selfie videos from Apple Photos, transcribes them, and builds a searchable knowledge base of your own thinking over time. The core philosophy: shift from passive consumption to active creation. Constantly creating rather than consuming.

## Project Phases

### Phase 0 — POC (Current Priority)
Validate the concept using existing tools before building custom infrastructure.

### Phase 1 — RAG Foundation (Weeks 1-6)
Upload → transcribe → chunk → embed → search your own thoughts.

### Phase 2 — Compiled Wiki Layer (Weeks 7-14)
Nightly Karpathy-style wiki compilation built ON TOP of RAG.

### Phase 3 — Reflection Engine + Behavioral Layer (Weeks 15-22)
Proactive synthesis, nudges, creation-over-consumption flywheel.

### Phase 4 — Scale + iOS App + Multi-Modal (Weeks 23+)
OpenSearch migration, native iOS companion, Apple Watch, CarPlay.

### How the POC Works

**Scan:** Uses `osxphotos` to read the Apple Photos database directly on macOS. Filters for `photo.selfie == True` (front-facing camera) + `media_type == video` + duration between 30s-20min. Outputs `selfie_videos.csv` for human review.

**Process:** For each video marked "yes" in the CSV:
1. Uploads video to Gemini File API (temporary, auto-deletes)
2. Gemini transcribes verbatim with timestamps, topic tags, open questions
3. Generates a summary + tags via a second Gemini call
4. Appends transcript to the appropriate monthly Google Doc
5. When all videos in a month are processed, generates a monthly overview

### Key Design Decisions

**pgvector over OpenSearch Serverless:** OpenSearch Serverless minimum ~$700/mo. pgvector on RDS PostgreSQL ~$30-50/mo on db.t4g.medium. Migrate to OpenSearch at Phase 4 if needed.

**Step Functions over Lambda chains:** Multi-step pipeline with failure modes at each stage. Retry logic, error handling, visibility for free.

**Batch transcription over real-time:** AWS Transcribe batch is cheaper, more accurate, better speaker diarization.

**Multi-tenancy model:** Every resource partitioned by `tenant_id` (Cognito user pool sub). S3 keys: `{tenant_id}/videos/{uuid}.mov`. DynamoDB: PK=tenant_id, SK=video_id. pgvector: filtered by tenant_id on every query.

### Transcript Cleaning Step
Before chunking, Claude pass that:
- Removes filler sounds (um, uh) while preserving meaning
- Fixes transcription errors using context
- Adds paragraph breaks at topic shifts
- Tags emotional indicators [excited], [uncertain]
- Preserves speaker's actual words (NOT rewriting)
- Cleaned transcript stored alongside raw — both kept

### Chunking Strategy for Spoken Language
Semantic chunking, NOT fixed-token:
- Split on topic boundaries (from Claude cleaning pass paragraph breaks)
- Each "thought block" = one chunk (30-90 seconds of speech)
- Overlap: last sentence of previous chunk as context prefix
- Metadata per chunk: tenant_id, video_id, timestamp_start/end, topic_tags, emotional_tone, chunk_sequence_number

### Data Model (PostgreSQL + pgvector)
```sql
CREATE TABLE tenants (
    tenant_id UUID PRIMARY KEY,
    cognito_sub TEXT UNIQUE NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now(),
    plan_tier TEXT DEFAULT 'free',
    wiki_enabled BOOLEAN DEFAULT false
);

CREATE TABLE videos (
    video_id UUID PRIMARY KEY,
    tenant_id UUID REFERENCES tenants(tenant_id),
    s3_key TEXT NOT NULL,
    duration_sec INTEGER,
    recorded_at TIMESTAMPTZ,
    uploaded_at TIMESTAMPTZ DEFAULT now(),
    status TEXT DEFAULT 'uploaded',
    raw_transcript TEXT,
    cleaned_transcript TEXT,
    summary TEXT,
    topic_tags TEXT[]
);

CREATE TABLE chunks (
    chunk_id UUID PRIMARY KEY,
    tenant_id UUID REFERENCES tenants(tenant_id),
    video_id UUID REFERENCES videos(video_id),
    chunk_index INTEGER NOT NULL,
    content TEXT NOT NULL,
    timestamp_start FLOAT,
    timestamp_end FLOAT,
    topic_tags TEXT[],
    emotional_tone TEXT,
    embedding vector(1024),
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX chunks_embedding_idx ON chunks
    USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
CREATE INDEX chunks_tenant_idx ON chunks(tenant_id);
```

### Query Flow
1. Embed query → Titan v2
2. pgvector search: top 20 chunks filtered by tenant_id
3. Rerank → pick top 8
4. Claude synthesis with system prompt: "You are helping the user explore their own past thinking. Synthesize, note evolution over time, cite specific recordings by date."
5. Return synthesis + source links (deep links to video timestamps)

## Phase 2 — Compiled Wiki Layer

### Why Both RAG and Wiki (Critical Architectural Decision)

RAG alone: scales linearly, preserves everything, mature tooling, but no compounding — every query starts from scratch.

Wiki alone: compounds knowledge, human-readable, resolves contradictions, but compilation costs blow up at scale, degrades with 5000+ articles, and at scale you need retrieval to find relevant pages during compilation (reinventing RAG).

**Decision: Use both. RAG as foundation, Wiki built on top.**
- Layer 1 (RAG): always present, full fidelity, cheap ingestion
- Layer 2 (Wiki): periodic compilation using RAG retrieval to find context
- Layer 3 (Query Router): routes queries to the right layer

### Wiki Compilation Pipeline
```
EventBridge (nightly per tenant)
    │
    ▼
Step Function:
    1. Fetch new chunks since last compilation
    2. For each chunk, retrieve related wiki pages via RAG
    3. Claude decides: update page? create page? add link? flag contradiction?
    4. Write updated pages to wiki_pages table + S3 markdown backup
    5. Lint pass: stale pages, orphans, contradictions, open questions
```

### Wiki Data Model
```sql
CREATE TABLE wiki_pages (
    page_id UUID PRIMARY KEY,
    tenant_id UUID REFERENCES tenants(tenant_id),
    slug TEXT NOT NULL,
    title TEXT NOT NULL,
    category TEXT,  -- 'topic', 'person', 'project', 'question'
    content_md TEXT NOT NULL,
    summary TEXT,
    linked_pages UUID[],
    source_chunks UUID[],
    contradiction_flags TEXT[],
    open_questions TEXT[],
    confidence FLOAT,
    embedding vector(1024),
    first_created TIMESTAMPTZ DEFAULT now(),
    last_compiled TIMESTAMPTZ DEFAULT now(),
    version INTEGER DEFAULT 1,
    UNIQUE(tenant_id, slug)
);

CREATE TABLE wiki_page_versions (
    version_id UUID PRIMARY KEY,
    page_id UUID REFERENCES wiki_pages(page_id),
    tenant_id UUID,
    version INTEGER,
    content_md TEXT,
    compiled_at TIMESTAMPTZ DEFAULT now(),
    source_chunks UUID[],
    change_summary TEXT
);
```

### Compilation Prompt (key design artifact)
```
System: You are a personal knowledge compiler. You maintain a wiki that
represents a person's evolving thinking, built entirely from their own
spoken words.

RULES:
- Write in third person: "Mike believes...", "Mike has been exploring..."
- Always cite recordings: [2024-03-15 recording]
- Show evolution: "Initially (Jan 2024), Mike felt X. By March, this shifted to Y."
- Flag contradictions in a ## Contradictions section
- End each page with ## Open Questions
- Use the user's own language, not your interpretation
- Cross-link aggressively: [[career-strategy]], [[bjj-coaching]]
```

### Query Router
```python
if intent == "factual_lookup":
    # "What did I say about X?" → RAG directly
elif intent == "synthesized_understanding":
    # "How has my thinking evolved?" → Wiki first, RAG fallback

    # "What have I been thinking about?" → Wiki browsable UI
elif intent == "contradiction_check":
    # "Am I being consistent?" → Wiki contradiction flags
```

## Photos API Integration Notes

**Apple Photos / iCloud:** No public API for cloud access. PhotoKit on-device is the path — can query for `.selfie` + `.video` media subtypes directly. For macOS POC, `osxphotos` library reads the Photos database.

**Google Photos API:** Read-only Library API exists but content-category filters deprecated in 2023. Can list all media and filter client-side.

**POC approach:** `osxphotos` on Mac → Gemini File API → Google Docs → NotebookLM.

**Production approach:** Native iOS app using PhotoKit to filter and upload directly to S3.

## Karpathy Wiki Pattern Reference

Andrej Karpathy's LLM Wiki (published April 3, 2026):
- Raw sources → LLM compilation → structured interlinked wiki
- Compiler analogy: source code (raw docs) → compiler (LLM) → executable (wiki)
- Three operations: Ingest, Query, Lint
- Single ingest can touch 10-15 wiki pages
- Key insight: "Knowledge is compiled once and continuously refined, not re-derived"
- Connected to Vannevar Bush's Memex (1945)
- His wiki grew to ~100 articles, ~400K words on a single topic

**How we adapted it:** Wiki compilation uses RAG retrieval to find relevant pages during compilation. This solves the scale problem (can't read entire wiki on every ingest). RAG is the foundation; wiki is the differentiated product layer built on top.

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| Car audio transcription quality | High | Test with real samples first (POC validates this). Gemini multimodal helps (sees video frames too). |
| Wiki compilation costs at scale | High | Rate-limit to nightly. Batch chunks. Per-tenant LLM spend caps. |
| pgvector at 1M+ chunks | Medium | Benchmark at 500K. Migration path to OpenSearch designed. |
| Users don't record consistently | High | Phase 3 nudge system. Accept product works for existing-habit users. |
| Compilation hallucinates content | High | Every wiki page cites source chunks. Lint cross-checks. User flagging. |
| Multi-tenant compilation thundering herd | Medium | Jitter schedules across 24hr window. Priority queue. |
